Background

Sharing and working in public is an ideal that I try to aspire to. Those are two different ideas worth exploring. Sharing in this context means sharing the great work of others, and helping them get more visibility in the small way that I can. Working in public, in this context, means sharing more of my own habits, routines, and the way I live, engage in the world. The blogroll does both of these. It shares writings of others, and it shares what I am reading and what I find interesting. It does so in an interesting way as well. That's what I'll break down here.

If you're already here wondering what the blogroll I'm referring to is, you should visit it here and come back. That being said, a blogroll is a list of links to blogs and websites updated using RSS/Atom feeds. It is a way to aggregate content from multiple sources into one place, allowing readers to curate their own personalized media feed. The inspiration for this blogroll comes from Alex Kladov's blog "matklad", specifically the post "RSS Server Side Reader". If you want to know more about RSS/Atom feeds, you should read their brief primer on it in that post. It explains it better than I could.

How it works

The goal of software is to abstract away complexity. I take the same approach when building my development environment and CI/CD pipelines. This allows for easy and reliable builds, and during development, it serves as a reproducible way to perform common tasks with minimal overhead.

My blogroll implementation has three main components: a build wrapper script that orchestrates everything, a Go binary that fetches RSS feeds and generates data, and a Hugo template that renders the final page. Let me walk through how they work together.

Build Wrapper Script

Hugo doesn't have built-in hooks for running arbitrary commands during the build process, so I created a wrapper script around hugo build using devenv. Devenv allows you to define your development environment declaratively in a devenv.nix file, including dependencies, scripts, processes, and precommit hooks.

The wrapper script preprocesses data before Hugo generates the production build. When I run hugo-build (locally or in CI), it triggers this call stack:

  1. hugo-build: preprocesses data and provides Hugo with additional build data
  2. hugo-update-blogroll: calls the Go blogroll generator
  3. blogroll-generator-run: ensures the binary is built and runs it
  4. blogroll-generator-build: builds the Go binary

This creates a blogroll.yaml data file that Hugo can use during its build process. All of these scripts can be called individually for development purposes, but the stack is designed to be simple to use in practice.

Go Code

The Go program is responsible for fetching RSS feeds from /static/assets/blogroll.txt, parsing them, and generating a YAML file that Hugo can use. The program is intended to be error resilient, if one feed is down or malformed, it logs the error and continues processing the rest rather than failing the entire build.

I deliberately keep the extracted data minimal: title, URL, date, domain, and feed URL. This gives readers enough context to decide what interests them without overwhelming the page. The program handles both RSS and Atom formats by attempting RSS parsing first, then falling back to Atom if that fails.

One of the more interesting aspects was date parsing. RSS and Atom feeds use wildly different date formats, so the parser tries nine different time formats before giving up on a post. The program also limits each feed to the three most recent posts, then sorts everything chronologically across all feeds to create a unified timeline.

The blogroll.txt file supports comments (lines starting with #) and automatically filters out empty lines, making it easy to organize and temporarily disable feeds without deleting URLs entirely.

Hugo Template

The Hugo template is where the blogroll comes to life visually. I wanted to create something that felt distinctly technical while remaining readable, so I designed it around a terminal aesthetic that would resonate with the developer audience likely to discover RSS feeds interesting.

The template starts with a simulated bat command output showing the contents of blogroll.txt. This serves both as transparency (showing exactly which feeds are being followed) and as a visual hook. The template deduplicates feed URLs for this display since multiple recent posts can come from the same feed, building up a unique list as it processes the data. The template handles the case where no blogroll data exists, showing helpful instructions to run the generator.

Below that, the posts are displayed in a clean table format with hover effects and external link indicators. I used semantic HTML throughout: proper table headers with scope attributes, time elements with datetime attributes, and descriptive aria-label text for screen readers.

Closing thoughts

Building this blogroll was as much about the process as the outcome. By sharing both what I read and how I built the tool, I'm hoping to embody both aspects of working in public: amplifying others' voices while being transparent about my own methods and choices.
